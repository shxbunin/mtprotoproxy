# AGENTS.md

## Purpose

This repository is an async MTProto proxy for Telegram written almost entirely in one file: `mtprotoproxy.py`.

The proxy accepts Telegram client connections, validates the client handshake against configured secrets, then forwards traffic to Telegram either:

- directly to Telegram data centers;
- through Telegram middle proxies when advertising is enabled.

This project is protocol-sensitive. Small byte-level changes can break client compatibility, replay protection, masking, or middle-proxy support.

## File Map

- `mtprotoproxy.py`: main application, protocol implementation, config normalization, network server, metrics, background tasks.
- `config.py`: user-facing configuration with port, secrets, modes, `TLS_DOMAIN`, `AD_TAG`.
- `Dockerfile`: container runtime with `python3`, `uvloop`, `cryptography`, `socks`.
- `docker-compose.yml`: host-network deployment with mounted `config.py` and `mtprotoproxy.py`.
- `pyaes/`: bundled fallback crypto backend used only when faster crypto libraries are unavailable.
- `README.md`: short startup guide.

## Core Architecture

Most logic lives in `mtprotoproxy.py` and is split conceptually into these layers:

- config loading and defaulting;
- crypto backend selection;
- connection pooling to Telegram;
- layered stream wrappers for TLS-like wrapping, AES wrapping, MTProto framing, and middle-proxy RPC framing;
- client handshake parsing and validation;
- direct Telegram connection path;
- middle-proxy connection path;
- traffic forwarding, per-user accounting, metrics, and background maintenance tasks.

There are almost no tests and almost no module boundaries. Treat the file as a protocol appliance, not as a typical app.

## How The Proxy Works

### Startup

- `main()` loads config from `config.py` or from a custom path.
- `init_config()` normalizes defaults and expands advanced settings not present in `config.py`.
- `apply_upstream_proxy_settings()` optionally monkey-patches `socket.socket` for upstream SOCKS5.
- `init_ip_info()` discovers external IPv4 and IPv6 addresses.
- `print_tg_info()` prints `tg://proxy` links for each configured user and enabled mode.
- Background tasks start for stats, mask-host probing, DNS cache clearing, Telegram middle-proxy updates, and time synchronization checks.
- Listeners are created for IPv4, IPv6, Unix socket, and optional metrics endpoint.

### Incoming Client Flow

1. `handle_client()` sets socket keepalive, buffer sizes, and ACK timeout.
2. The client must complete handshake within `CLIENT_HANDSHAKE_TIMEOUT`.
3. If `PROXY_PROTOCOL=True`, the proxy first expects valid PROXY protocol v1 or v2 headers.
4. The code then tries to detect a fake TLS handshake.
5. If the connection is TLS-like, `handle_fake_tls_handshake()` validates the HMAC-based digest using each configured user secret.
6. The TLS path also checks timestamp skew and replayed handshakes.
7. If TLS validation succeeds, the transport is wrapped in `FakeTLSStreamReader` and `FakeTLSStreamWriter` and the internal MTProto handshake is read.
8. If the connection is not TLS-like, only `classic` and `secure` modes are allowed.
9. The internal 64-byte MTProto handshake is decrypted with candidate user secrets until a valid `proto_tag` is found.
10. The code extracts the MTProto transport type and Telegram data center index.
11. Replay protection is applied again for the non-TLS path.
12. The client stream is wrapped in AES-CTR crypto wrappers.

### Routing Decision

After successful client handshake the proxy chooses one of two upstream paths:

- direct path: used when `USE_MIDDLE_PROXY=False` or middle-proxy mode was disabled at runtime;
- middle-proxy path: used when advertising is enabled and middle-proxy mode is active.

### Direct Path

- Connects straight to one of Telegram's hardcoded data centers on port `443`.
- Builds a fresh obfuscated MTProto handshake for Telegram.
- Filters out reserved nonce prefixes so the generated bytes do not look like HTTP, TLS, or known protocol signatures.
- In `FAST_MODE`, the Telegram-to-client direction skips re-encryption by reusing handshake key material. This is faster but intentionally less secure.

### Middle Proxy Path

- Connects to a Telegram middle proxy chosen from periodically refreshed proxy lists.
- Performs a separate RPC-based middle-proxy handshake.
- Derives AES-CBC keys from nonce, IP, port, direction, and Telegram's middle-proxy secret.
- Wraps traffic in `ProxyReqStreamReader` and `ProxyReqStreamWriter`.
- Injects `AD_TAG` into proxy requests, which is how Telegram channel advertising works.
- Applies client-side MTProto packet framing depending on `abridged`, `intermediate`, or `secure` transport.

### Traffic Forwarding

- Two async loops move data in both directions.
- User stats are tracked per direction: octets, messages, connects, current connects.
- Telegram read timeout is enforced on server-to-client direction.
- When either direction ends, both forwarding tasks are cancelled and the upstream transport is aborted.

### Bad Clients And Masking

If the handshake is invalid:

- `connects_bad` is incremented;
- if `MASK=False` or there is no captured handshake, the proxy just drains input and closes;
- if `MASK=True`, the proxy opens a connection to `MASK_HOST:MASK_PORT`, forwards the original handshake there, and relays data both ways.

This masking behavior is part of the anti-detection design. It should not be simplified casually.

## Supported Modes

### `classic`

- Non-TLS mode.
- Accepts classic MTProto transports such as `abridged` and `intermediate`.
- Easiest mode to detect.
- Proxy links use the raw secret.

### `secure`

- Non-TLS mode using secure MTProto transport.
- Harder to detect than `classic`.
- Can be incompatible with very old clients.
- Proxy links use `dd` + secret.

### `tls`

- Fake TLS on the outside, secure MTProto inside.
- Hardest mode to detect in this implementation.
- Depends on HMAC-validated fake TLS handshake, time checks, and a believable `TLS_DOMAIN`.
- Can be incompatible with older clients.
- Proxy links use `ee` + secret + hex-encoded `TLS_DOMAIN`.

## Direct vs Middle Proxy

### Direct

- Simpler path.
- No advertising.
- Can use `FAST_MODE`.
- Uses Telegram data center IP lists embedded in the source.

### Middle Proxy

- Required for Telegram advertising via `AD_TAG`.
- Depends on Telegram's middle-proxy config and secret fetched at runtime.
- Uses custom RPC framing and AES-CBC handshake.
- More fragile operationally than direct mode.

## Important Config Keys

### Identity And Access

- `USERS`: mapping of username to 32-hex-character secret.
- `PORT`: listen port.
- `MODES`: enables `classic`, `secure`, `tls` independently.

### TLS And Masking

- `TLS_DOMAIN`: domain encoded into TLS proxy links and used for mask-host probing.
- `MASK`: whether bad clients should be proxied to a real host.
- `MASK_HOST`: destination for bad clients.
- `MASK_PORT`: destination port for bad clients.
- `MY_DOMAIN`: optional public domain shown in generated links instead of detected IP.

### Advertising And Upstream Behavior

- `AD_TAG`: 16-byte tag from `@MTProxybot`.
- `USE_MIDDLE_PROXY`: whether to route through Telegram middle proxies.
- `FAST_MODE`: direct mode optimization that disables Telegram-to-client re-encryption.
- `PREFER_IPV6`: prefer IPv6 for Telegram connections when possible.
- `SOCKS5_HOST`, `SOCKS5_PORT`, `SOCKS5_USER`, `SOCKS5_PASS`: upstream SOCKS5 settings.

### Safety And Abuse Control

- `REPLAY_CHECK_LEN`: replay cache size. `0` disables replay protection.
- `IGNORE_TIME_SKEW`: accepts clients with bad clocks, weakening replay defenses.
- `USER_MAX_TCP_CONNS`: per-user TCP connection limit.
- `USER_EXPIRATIONS`: per-user expiration date.
- `USER_DATA_QUOTA`: per-user traffic quota.

### Network And Observability

- `PROXY_PROTOCOL`: require PROXY protocol v1 or v2 on incoming connections.
- `LISTEN_ADDR_IPV4`, `LISTEN_ADDR_IPV6`, `LISTEN_UNIX_SOCK`: listener addresses.
- `METRICS_PORT`: enables Prometheus-compatible metrics endpoint.
- `METRICS_WHITELIST`: IP allowlist for metrics.
- `STATS_PRINT_PERIOD`: interval for periodic console stats.

## Real Restrictions And Incompatibilities

- Upstream SOCKS5 is incompatible with middle-proxy advertising mode.
- Upstream SOCKS5 is also incompatible with `uvloop` in this codebase.
- IPv6 is considered unstable for middle proxies, so defaults prefer IPv4 in that mode.
- `secure` and `tls` may break older Telegram clients.
- Middle-proxy mode can be disabled automatically at runtime if server time is too skewed.
- If the proxy cannot determine its public IP, middle-proxy advertising is disabled.
- `MASK_HOST` should ideally behave like a TLS 1.3 host. Non-TLS-1.3 behavior is explicitly marked as not recommended.
- A mask host that emits an unexpected TLS record pattern makes the proxy easier to detect.
- When `PROXY_PROTOCOL=True`, non-compliant clients are treated as bad clients immediately.
- There is no automated test suite in this repository.

## Security Model And Tradeoffs

- Handshakes are authenticated against per-user secrets.
- Replay protection exists for both TLS and non-TLS handshake paths.
- TLS mode validates client time skew to make replay attacks harder.
- `IGNORE_TIME_SKEW=True` weakens replay protection.
- `FAST_MODE=True` improves throughput but intentionally reduces confidentiality on the Telegram-to-client path.
- Masking invalid clients to a believable host is part of the anti-detection story.

## Invariants And Prohibitions

These are the most important project-specific rules for future agents and maintainers.

- Do not change handshake lengths or offsets casually.
- Do not change `SKIP_LEN`, `PREKEY_LEN`, `KEY_LEN`, `IV_LEN`, `HANDSHAKE_LEN`, `PROTO_TAG_POS`, or `DC_IDX_POS` without a full byte-level protocol review.
- Do not modify MTProto framing classes unless you understand the on-wire format end to end.
- Do not assume `classic`, `secure`, and `tls` are interchangeable. Each mode affects handshake parsing, link generation, and framing.
- Do not enable or preserve middle-proxy behavior together with upstream SOCKS5.
- Do not remove replay checks, time-skew checks, or bad-client masking without an explicit reason.
- Do not silently change the meaning of existing config keys.
- Do not treat `FAST_MODE` as a harmless optimization. It changes security properties.
- Do not assume failures will be loud. This code contains many silent fallbacks and `return False` paths.
- Do not break the Windows-specific event-loop branch.
- Do not assume metrics are public. The metrics endpoint is meant to stay IP-restricted.

## Risky Areas For Changes

Highest risk:

- `handle_fake_tls_handshake()`
- `handle_handshake()`
- `do_direct_handshake()`
- `get_middleproxy_aes_key_and_iv()`
- `middleproxy_handshake()`
- `ProxyReqStreamReader` and `ProxyReqStreamWriter`
- MTProto framing reader and writer classes

Medium risk:

- `handle_bad_client()`
- `apply_upstream_proxy_settings()`
- `init_config()`
- `update_middle_proxy_info()`
- `get_mask_host_cert_len()`

Lower risk:

- metrics formatting and exporting;
- console stats printing;
- non-protocol config defaults;
- Docker packaging.

## Safe Editing Guidelines

- Make the smallest possible change.
- Prefer preserving byte layout and control flow in handshake code.
- If a refactor is needed, first extract pure helpers away from protocol reads and writes.
- Preserve all runtime incompatibility guards unless the change is deliberate and documented.
- Be careful with shared global state such as `used_handshakes`, `client_ips`, `disable_middle_proxy`, `fake_cert_len`, and dynamic proxy lists.
- Remember that this repository may run on both Linux and Windows.

## How To Run

Directly:

```bash
python3 mtprotoproxy.py
```

With custom config:

```bash
python3 mtprotoproxy.py path/to/config.py
```

With Docker Compose:

```bash
docker-compose up -d
```

Config reload at runtime is supported through `SIGUSR2` on platforms that have it.

## Suggested Manual Verification

Because there are no tests, protocol-sensitive changes should be verified manually.

- start proxy and confirm generated `tg://proxy` links look correct;
- verify `classic` mode connection;
- verify `secure` mode connection;
- verify `tls` mode connection with a real `TLS_DOMAIN`;
- verify direct path;
- verify middle-proxy path when `AD_TAG` is configured;
- verify bad-client masking still works;
- verify metrics endpoint only responds to whitelisted IPs;
- verify SOCKS5 mode still disables middle proxy and avoids `uvloop`.

## Practical Notes

- Default secret values and default `TLS_DOMAIN` are intentionally warned against at startup.
- Port `3256` is the fallback default but is not recommended.
- `AD_TAG` of length 16 implicitly enables middle-proxy mode unless overridden.
- User limits are enforced after connection setup begins, so behavior around limits is closer to fast termination than early rejection.
- Dependency reality is defined more by `Dockerfile` than by packaging metadata because there is no `requirements.txt`.
